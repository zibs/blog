The consensus is that Rack is two things:
1.Rack is a ruby gem. Specifically, Rack is a gem built atop Ruby's Net:HTTP library. Net:HTTP is the HTTP client API for ruby. As such, Rack provides an interface for using this HTTP API in Ruby. This brings us to...
2.Rack is a web-server interface convention. In simpler terms: "Rack provides a minimal interface between webservers that support Ruby and Ruby frameworks." This minimal interface enables easier communications between rack-compliant web-servers.

This makes its easy to build extremely simple web applications in ruby using rack (although, both Sinatra and Rails use rack as well). Rack's implementation is also a form of the CS concept of Middleware. At its most basic level, middleware here refers to the way rack provides "a way to filter requests and responses". These filters can involve authentication (passing login parameters to methods which can authenticate a user), authorization, caching etc. In this case, a layer of middleware can be swapped in (with the use keyword) to perform these operations, and return the necessary three element array along with the result.

At a higher level, Rack's middleware implements a pipeline design pattern which emphasizes a separation of concerns. Middleware can be thought of constituting a middle ground for interactions between a client and server. A request comes in, middleware intercepts the request, reads its requirements (http headers, statuses etc) and then directs it to the appropriate place for a response.
