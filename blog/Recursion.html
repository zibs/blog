Recursion is awesome. Recursion is. Recursion. Recursion is simply the process of a function repeatedly applying itself to a set of problems until some base condition is met. In other words, recursion is the repeated application of a task on a problem until a specified state is reached. To achieve this, we first define what's known as the base condition which will terminate the recursive loop that is set into motion after its definition. Then we simply define a set of repeatable processes which will reduce all other cases to the base case by using the same function. For example:

In both we can see the base condition defined at the beginning of the function and then we see the recursive, repeatable process that is taken in order to eventually end up at that base condition. In the former, we traverse the string until it's empty, adding the first character to the first character of the rest of the characters until its finished, and in the latter, we compare the first element with the last element and then proceed recursively, passing in the array range which resulted in the larger number from the initial comparison.

The next example uses recursion to perform a binary search. A binary search takes a sorted array, goes to middle element and tests if the number is greater than or lesser than the number being searched for. If the number is greater, it ignores the first half of the array - or vice versa - and performs this operation recursively until the number is found.

The reason that recursion, or a binary search in this case, might be useful is in terms of performance. A binary search operation is O(log N) at its worst case. A sequential search is O(N). The difference is that to search through a billion items using linear, or sequential search (a loop), versus using binary serach is the difference between eleven and a half days vs 30 milliseconds. This is a great example of when to use recursion.

Although recursion is not always the most performant operation, we can see that here with something like factorial recursion it is. The performance of recursion depends on the type of task is to be performed, as well as what language we're using, and if that language performs Tail Call Optimization (which ruby doesn't by default.) Ruby has a limited amount of stack memory and so will error out with a SystemStackError: stack level too deep if the method call is too complex (like say if we try recursive_factorial_tutorial(110000)). This means that Ruby continuously piles method calls onto the stack, while with TCO or Tail Call Elimination, a function which returns itself as the final step of the function does not create a new stack frame, thus saving space and so optimizing recursion.
